## 2. 鉴权方式

接口支持两种鉴权方式：

1. SDK 调用：通过 `Authorization: Bearer xxx` 鉴权。
2. 非 SDK 调用：通过时间戳、accessKey、HMAC-SHA256 签名鉴权。

---

### 2.1 方式一：SDK 调用

请求头中携带：

```http
Authorization: Bearer xxx
```

说明：

* `xxx` 为具体秘钥。
* 具体秘钥需私下联系接口维护人获取。
* 注意 `Bearer` 与 token 之间需要有空格。

---

### 2.2 方式二：非 SDK 调用

非 SDK 调用时，需要基于时间戳和密钥生成签名。

示例逻辑：

```javascript
var ts = Date.now();

var key = "xxx";

var accessKey = "xxx";

var digestHash = CryptoJS.HmacSHA256(ts.toString(), key);

var digest = CryptoJS.enc.Base64.stringify(digestHash);

pm.environment.set("accessKey", accessKey);

pm.environment.set("ts", ts);

pm.environment.set("sign", digest);
```

字段含义：

| 字段           | 说明                                 |
| ------------ | ---------------------------------- |
| `ts`         | 当前时间戳，通常使用 `Date.now()` 获取         |
| `key`        | 用于 HMAC-SHA256 签名的密钥               |
| `accessKey`  | 访问标识                               |
| `digestHash` | 使用 `ts` 和 `key` 生成的 HMAC-SHA256 结果 |
| `digest`     | Base64 编码后的签名                      |
| `sign`       | 最终传递的签名字段                          |

## 3. 接口整体请求结构

接口整体风格与 OpenAI Chat Completions 接口类似，核心请求字段如下：

```json
{
  "model": "XIAOYI_DEEPSEEK_V3.1",
  "messages": [
    {
      "role": "user",
      "content": "你好"
    }
  ],
  "stream": true,
  "stream_options": {
    "include_usage": true
  },
  "temperature": 0.3,
  "top_p": 1.0,
  "top_k": 1,
  "max_tokens": 1024,
  "extra_body": {
    "call_from": "default",
    "session_id": "session_xxx",
    "interaction_id": 1,
    "enable_thinking": true
  }
}
```

## 4. 顶层请求字段说明

## 6. Messages 字段说明

`messages` 用于传递调用上下文，是一个数组。每个元素表示一条消息。

### 6.1 Message 基本结构

```json
{
  "role": "user",
  "content": "你好"
}
```

### 6.2 Message 字段

| 字段           |        类型 | 是否必选 | 说明                                                                           |
| ------------ | --------: | ---: | ---------------------------------------------------------------------------- |
| `role`       |    String |    是 | 消息角色，取值范围：`system`、`user`、`assistant`、`tool`                                 |
| `content`    |    String |    是 | 消息文本内容                                                                       |
| `reasoning`  |    String |    否 | 当 `role=assistant` 时，可填入大模型思考文本                                              |
| `tool_calls` | JSONArray |    否 | 当 `role=assistant` 时，表示当前 message 为 function call 结果，仅支持 function call 的模型生效 |

### 6.3 Role 取值说明

| role        | 说明              |
| ----------- | --------------- |
| `system`    | 系统指令，用于约束模型整体行为 |
| `user`      | 用户输入            |
| `assistant` | 模型回复            |
| `tool`      | 工具调用结果          |

## 7. Tools / Function Call 字段说明

`tools` 用于指定可供模型调用的工具列表。模型会根据上下文选择是否调用其中一个工具。

### 7.1 支持 Function Call 的现网模型

| 模型名称                           |
| ------------------------------ |
| `XIAOYI_DEEPSEEK_V3.1`         |
| `AGENT-DEEPSEEK-V3.1`          |
| `AGENT-DEEPSEEK-V3.1-THINKING` |
| `AGENT-DEEPSEEK-V3.2`          |
| `AGENT-DEEPSEEK-V3.2-THINKING` |

### 7.2 现网暂未支持 Function Call 的模型

| 模型名称                | 说明                          |
| ------------------- | --------------------------- |
| `XIAOYI_PANGU_718B` | PANGU-718B-SFT 大模型，10 月 OBP |
| `LLM_DEEPSEEK_V3.2` | DSV3.2，10 月 OBP             |
| `LLM_DEEPSEEK_V3.1` | DSV3.1，10 月 OBP             |

### 7.3 Tools 字段结构

```json
{
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "search_memory",
        "description": "用于检索用户历史记忆",
        "parameters": {
          "type": "object",
          "properties": {
            "query": {
              "type": "string",
              "description": "检索关键词"
            }
          },
          "required": ["query"]
        }
      }
    }
  ]
}
```

### 7.4 Tools 子字段说明

| 字段                             |         类型 | 说明                        |
| ------------------------------ | ---------: | ------------------------- |
| `tools[].type`                 |     String | 工具类型，当前仅支持 `function`     |
| `tools[].function`             | JSONObject | function 详情               |
| `tools[].function.name`        |     String | 工具函数名称                    |
| `tools[].function.description` |     String | 工具函数描述，供模型判断何时以及如何调用      |
| `tools[].function.parameters`  | JSONObject | 工具参数描述，通常为 JSON Schema 格式 |

## 8. Stream 字段说明

### 8.1 stream

| 字段       |      类型 | 是否必选 |    默认值 | 说明       |
| -------- | ------: | ---: | -----: | -------- |
| `stream` | Boolean |    否 | `true` | 是否启用流式输出 |

说明：

* `stream=true`：接口以流式分块形式返回。
* `stream=false`：接口一次性返回完整结果。
* 该接口默认启用流式输出。

如果业务代码没有处理 SSE/流式响应，建议显式设置：

```json
{
  "stream": false
}
```

---

### 8.2 stream_options

`stream_options` 仅在 `stream=true` 时有效。

```json
{
  "stream_options": {
    "include_usage": true
  }
}
```

| 字段                             |      类型 | 是否必选 |    默认值 | 说明                                         |
| ------------------------------ | ------: | ---: | -----: | ------------------------------------------ |
| `stream_options.include_usage` | Boolean |    否 | `true` | 是否在流式消息最后的 `data: [DONE]` 之前额外返回一个 usage 块 |

说明：

* 当 `include_usage=true` 时，最后会额外返回一个包含 token 使用统计的块。
* 该 usage 块中：

  * `usage` 字段显示整个请求的 token 使用情况；
  * `choices` 字段始终为空数组；
  * 其他流式分块也会包含 `usage` 字段，但值可能为 `null`。

## 9. 生成参数说明

| 字段            |        类型 | 是否必选 |  默认值 | 说明                               |
| ------------- | --------: | ---: | ---: | -------------------------------- |
| `temperature` |     Float |    否 | 依赖模型 | 控制生成内容的随机性，0 表示确定，2 表示极随机        |
| `top_p`       |     Float |    否 | 依赖模型 | 核采样阈值，仅从累计概率达到 Top-P 的 token 中选择 |
| `top_k`       |   Integer |    否 | 依赖模型 | 限制采样候选 token 数量                  |
| `max_tokens`  |   Integer |    否 | 依赖模型 | 生成响应的最大 token 数，包含模型回复内容         |
| `stop`        | JSONArray |    否 | `NA` | 停止生成的 token 序列，仅部分模型支持           |

示例：

```json
{
  "top_p": 1.0,
  "top_k": 1,
  "temperature": 0.3
}
```

## 10. Prompt Cache 字段说明

接口中存在两个位置可以传递 `promptCacheId`：

1. 顶层字段：`promptCacheId`
2. `extra_body.promptCacheId`

### 10.1 字段说明

| 字段                         |     类型 | 是否必选 |  默认值 | 说明                      |
| -------------------------- | -----: | ---: | ---: | ----------------------- |
| `promptCacheId`            | String |    否 | `NA` | Prompt 缓存唯一 ID          |
| `extra_body.promptCacheId` | String |    否 | `NA` | 业务扩展字段中的 Prompt 缓存唯一 ID |

### 10.2 使用注意事项

`promptCacheId` 需要满足以下要求：

1. value 需要符合 HTTP Header 规范。
2. 字符取值范围：`0~9`、`a~z`、`-`、`_`。
3. 在同一任务、同一多轮对话内保持一致。
4. 不同任务、不同多轮对话内尽量不要重复。
5. 多轮对话结束后需要及时清理。

## 11. Extra Body 业务字段说明

`extra_body` 用于隔离业务字段，建议业务侧自定义参数统一放到该对象中。

### 11.1 extra_body 示例

{
  "extra_body": {

## 13. 风控字段说明

风控字段位于：

```json
{
  "extra_body": {
    "moderation_options": {
      "action": "dudu"
    }
  }
}
```

### 13.1 moderation_options

| 字段                                     |         类型 | 是否必选 | 说明                       |
| -------------------------------------- | ---------: | ---: | ------------------------ |
| `extra_body.moderation_options`        | JSONObject |    否 | 风控处理信息，不传 `action` 时不做风控 |
| `extra_body.moderation_options.action` |     String |    否 | 风控 action，当前取值范围：`dudu`  |

### 13.2 风控策略说明

原文说明：

* 文本显示统一先审后发。
* 首 token 数大于等于 10。
* 流式子句策略 token 数最小 10，最大 15。
* 有标点符号时，满足最小 token 数即可。
* 没有标点符号时，需要满足 15 个 token 数。
* 标点符号范围包括：

```text
; , . ? ! ； ， 。 ？ ！ \n
```

## 14. 返回报文字段说明

接口返回分为两种情况：

1. `stream=true`：流式分块返回。
2. `stream=false`：一次性返回完整数据。

---

### 14.1 通用返回字段

| 字段名       |         类型 | 描述                                                 |
| --------- | ---------: | -------------------------------------------------- |
| `id`      |     String | 整个流请求的唯一标识，所有分块相同。返回请求报文中的 `extra_body.session_id` |
| `object`  |     String | 固定为 `chat.completion.chunk`，表示流模式分块                |
| `created` |       long | 请求创建时间戳，Unix 秒级                                    |
| `model`   |     String | 本次调用的模型名称                                          |
| `choices` |  JSONArray | 生成内容结果，当前仅一个元素                                     |
| `usage`   | JSONObject | 整个请求的 token 使用统计信息                                 |
| `error`   | JSONObject | 服务异常、模型异常、请求异常等场景下的报错字段                            |

---

### 14.2 choices 字段

| 字段名                        |         类型 | 描述                                                    |
| -------------------------- | ---------: | ----------------------------------------------------- |
| `choices[0].index`         |    Integer | 当前结果序号                                                |
| `choices[0].finish_reason` |     String | 生成状态                                                  |
| `choices[0].delta`         | JSONObject | `stream=true` 时的增量内容对象                                |
| `choices[0].message`       | JSONObject | `stream=false` 时的全量内容对象；部分流式 chunk 中也可能保留空 message 结构 |

---

### 14.3 finish_reason 取值说明

| 取值               | 说明                                  |
| ---------------- | ----------------------------------- |
| 空字符串或 `null`     | 生成中                                 |
| `stop`           | 模型自然停止生成，或遇到 `stop` 序列中列出的字符串       |
| `length`         | 输出长度达到模型上下文长度限制，或达到 `max_tokens` 限制 |
| `content_filter` | 输出内容因触发过滤策略而被过滤                     |
| `aborted`        | 其他异常场景，内部错误终止                       |

---

## 15. stream=true 返回字段

当 `stream=true` 时，返回内容主要位于：

```text
choices[0].delta
```

### 15.1 delta 字段说明

| 字段名                                  |        类型 | 描述                                         |
| ------------------------------------ | --------: | ------------------------------------------ |
| `choices[0].delta.role`              |    String | 模型角色，通常为 `assistant`                       |
| `choices[0].delta.content`           |    String | 模型回复的增量正文内容                                |
| `choices[0].delta.reasoning`         |    String | 模型增量推理内容                                   |
| `choices[0].delta.reasoning_content` |    String | 模型增量推理内容，思考过程                              |
| `choices[0].delta.tool_calls`        | JSONArray | 如果模型需要调用工具，则会生成该字段，仅支持 function call 的模型生效 |

### 15.2 delta.tool_calls 字段说明

| 字段名                                                 |         类型 | 描述                    |
| --------------------------------------------------- | ---------: | --------------------- |
| `choices[0].delta.tool_calls[i].type`               |     String | 工具类型，当前仅支持 `function` |
| `choices[0].delta.tool_calls[i].function`           | JSONObject | function 详情           |
| `choices[0].delta.tool_calls[i].function.name`      |     String | 工具函数名称                |
| `choices[0].delta.tool_calls[i].function.arguments` |     String | 模型生成的工具入参，string 类型   |

## 16. stream=false 返回字段

当 `stream=false` 时，返回内容主要位于：

```text
choices[0].message
```

其格式与 `choices[0].delta` 基本一致。

示意结构：

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "模型回复内容",
        "reasoning_content": "模型思考内容"
      },
      "finish_reason": "stop"
    }
  ]
}
```

---

## 17. Usage 字段说明

`usage` 表示 token 使用统计。

| 字段名                                         |      类型 | 描述                                        |
| ------------------------------------------- | ------: | ----------------------------------------- |
| `usage.prompt_tokens`                       | integer | 用户输入文本转换成 token 后的长度                      |
| `usage.completion_tokens`                   | integer | 模型生成回复转换成 token 后的长度                      |
| `usage.total_tokens`                        | integer | `prompt_tokens` 与 `completion_tokens` 的总和 |
| `usage.prompt_tokens_details.cached_tokens` | integer | 命中缓存的 prompt token 数，部分返回中可能出现            |

---

## 18. Error 字段说明

当服务异常、模型异常、请求异常时，返回 `error` 字段。

| 字段名             |     类型 | 描述   |
| --------------- | -----: | ---- |
| `error.message` | String | 报错信息 |

示例错误：

```text
request_body call_from wrong
```

含义：

* `call_from` 不在白名单中。
* 需要检查 `extra_body.call_from` 是否填写正确，或确认该业务标识是否已配置白名单。

## 19.1 流式调用：V3.1 / V3.2 不思考

说明：

* 这类模型通过模型名区分是否思考。
* 不思考版本不需要传 `enable_thinking`。
* 如需使用 V3.2，将 `model` 从 `AGENT-DEEPSEEK-V3.1` 替换为 `AGENT-DEEPSEEK-V3.2` 即可。

### 19.1.2 请求体示例

```json

  "messages": [
    {
      "role": "user",
      "content": "1+1等于几"
    }

  "max_tokens": 1000,
  "stream": true,
  "extra_body": {
    "session_id": "00558007",
    "interaction_id": 1,
    "call_from": "xxx"
  }
}
```

字段说明：

| 字段                          | 说明                                                              |
| --------------------------- | --------------------------------------------------------------- |
| `model`                     | 模型名称。`AGENT-DEEPSEEK-V3.1` 默认不思考；V3.2 可使用 `AGENT-DEEPSEEK-V3.2` |
| `messages`                  | 对话上下文                                                           |
| `max_tokens`                | 最大生成 token 数                                                    |
| `stream`                    | 是否启用流式返回。这里为 `true`                                             |
| `extra_body.session_id`     | 会话 ID，用于问题定位和日志串联                                               |
| `extra_body.interaction_id` | 轮次 ID                                                           |
| `extra_body.call_from`      | 垂域标识，受白名单控制                                                     |

注意：

* `call_from` 需要在白名单中，否则可能返回 `request_body call_from wrong`。
* `session_id` 会体现在流式返回体的 `id` 字段中。

### 19.1.3 返回体示例

```text

```

```text

```

## 19.2 流式调用：V3.1 / V3.2 思考

### 19.2.1 适用模型

| 模型                             | 说明        |
| ------------------------------ | --------- |
| `AGENT-DEEPSEEK-V3.1-THINKING` | V3.1，默认思考 |
| `AGENT-DEEPSEEK-V3.2-THINKING` | V3.2，默认思考 |

说明：

* 这类模型通过模型名区分是否开启思考。
* 思考版本不需要额外传 `enable_thinking`。
* 如需使用 V3.2，将 `model` 从 `AGENT-DEEPSEEK-V3.1-THINKING` 替换为 `AGENT-DEEPSEEK-V3.2-THINKING` 即可。

### 19.2.2 请求体示例

```json
{
  "model": "AGENT-DEEPSEEK-V3.1-THINKING",
  "messages": [
    {
      "role": "user",
      "content": "1+1等于几"
    }
  ],
  "max_tokens": 1000,
  "stream": true,
  "extra_body": {
    "session_id": "00558007",
    "interaction_id": 1,
    "call_from": "xxx"
  }
}
```

字段说明：

| 字段                          | 说明                                                                               |
| --------------------------- | -------------------------------------------------------------------------------- |
| `model`                     | 模型名称。`AGENT-DEEPSEEK-V3.1-THINKING` 默认思考；V3.2 可使用 `AGENT-DEEPSEEK-V3.2-THINKING` |
| `messages`                  | 对话上下文                                                                            |
| `max_tokens`                | 最大生成 token 数                                                                     |
| `stream`                    | 是否启用流式返回。这里为 `true`                                                              |
| `extra_body.session_id`     | 会话 ID                                                                            |
| `extra_body.interaction_id` | 轮次 ID                                                                            |
| `extra_body.call_from`      | 垂域标识，受白名单控制                                                                      |

### 19.2.3 返回体示例

思考模型的流式返回中，除正文 `content` 外，还可能返回思考字段：

```text
data: {"id":"00558007","object":"chat.completion.chunk","created":1771903462,"model":"AGENT-DEEPSEEK-V3.1-THINKING","choices":[{"index":0,"message":{"role":"","content":"","reasoning":"","reasoning_content":"","tool_calls":null},"delta":{"role":"assistant","content":"","reasoning":"嗯","reasoning_content":"嗯","tool_calls":null},"finish_reason":""}],"usage":{"completion_tokens":1,"prompt_tokens":9,"prompt_tokens_details":{"cached_tokens":8},"total_tokens":10}}
```

### 19.2.4 返回体解析说明

思考模型返回时，客户端需要区分两个字段：

| 字段                                   | 说明                                   |
| ------------------------------------ | ------------------------------------ |
| `choices[0].delta.reasoning_content` | 增量思考内容                               |
| `choices[0].delta.reasoning`         | 增量思考内容，当前示例中与 `reasoning_content` 一致 |
| `choices[0].delta.content`           | 增量正文内容                               |

一般解析方式：

1. 将所有 chunk 中的 `delta.reasoning_content` 拼接为思考内容。
2. 将所有 chunk 中的 `delta.content` 拼接为最终回复正文。
3. 当 `finish_reason=stop` 或收到 `data: [DONE]` 时，表示流式响应结束。

示意伪代码：

```python
reasoning_text = ""
answer_text = ""

for chunk in stream_chunks:
    delta = chunk["choices"][0]["delta"]

    reasoning_text += delta.get("reasoning_content") or ""
    answer_text += delta.get("content") or ""

    finish_reason = chunk["choices"][0].get("finish_reason")
    if finish_reason == "stop":
        break
```

注意：

* 思考内容和最终回复内容是两个不同字段。
* 业务侧是否展示思考内容，需要根据产品策略决定。
* 若只需要最终答案，只拼接 `delta.content` 即可。
* 若需要记录模型思考过程，可额外拼接 `delta.reasoning_content`。

## 19.3 流式返回通用解析规则

无论是否思考，流式响应都可以按以下规则解析。

### 19.3.1 SSE 数据格式

流式返回一般形如：

```text
data: {...}

data: {...}

data: [DONE]
```

其中：

| 内容             | 说明            |
| -------------- | ------------- |
| `data: {...}`  | 一个 JSON chunk |
| `data: [DONE]` | 流式响应结束标记      |

### 19.3.2 正文拼接

非思考模型或只关心最终答案时，拼接：

```text
choices[0].delta.content
```

### 19.3.3 思考内容拼接

思考模型中，若需要获取思考内容，拼接：

```text
choices[0].delta.reasoning_content
```

### 19.3.4 结束判断

通常可以通过两种方式判断结束：

方式一：判断是否收到：

```text
data: [DONE]
```

方式二：判断最后一个 JSON chunk 中是否出现：

```json
"finish_reason": "stop"
```

其他 `finish_reason` 取值也需要关注：

| finish_reason    | 说明                       |
| ---------------- | ------------------------ |
| `stop`           | 正常结束                     |
| `length`         | 达到上下文长度或 `max_tokens` 限制 |
| `content_filter` | 触发内容过滤                   |
| `aborted`        | 内部异常终止                   |
| 空字符串或 `null`     | 生成中                      |

## 21. 接入注意事项

### 21.1 模型选择

* 需要明确区分测试环境和现网环境支持的模型。
* 部分测试模型目前只在测试环境可用，例如：

  * `kimik25`
  * `GLM-4.7`
  * `glm-5`
  * `qwen3.5`
* `AGENT-DEEPSEEK-*-THINKING` 类模型通过模型名控制是否思考，不建议再额外传 `enable_thinking`。
* `XIAOYI_DEEPSEEK_V3.1` 等支持快慢思考的模型，可通过 `extra_body.enable_thinking` 控制。

### 21.2 Stream 默认值

* `stream` 默认值为 `true`。
* 如果业务代码没有处理 SSE/流式响应，建议显式设置：

```json
{
  "stream": false
}
```

### 21.3 call_from

* `extra_body.call_from` 用于垂域标识和时延监控。
* 不传时默认为 `default`。
* 如果传入的 `call_from` 不在白名单内，可能返回：

```text
request_body call_from wrong
```

### 21.4 session_id 与 interaction_id

建议每次请求都传：

```json
{
  "session_id": "xxx",
  "interaction_id": 1
}
```

用途：

* 定位线上问题。
* 串联上下游日志。
* 辅助排查一次多轮对话中的具体轮次。

### 21.5 Prompt Cache

使用 `promptCacheId` 时需要注意：

* 同一任务、同一多轮内保持一致。
* 不同任务、不同多轮内尽量不重复。
* 多轮结束后及时清理。
* 字符范围需符合规范：`0~9`、`a~z`、`-`、`_`。

### 21.6 Function Call

使用 function call 时需要注意：

* 只有部分模型支持。
* `tools[].type` 当前仅支持 `function`。
* 工具参数建议使用标准 JSON Schema 描述。
* 模型返回的 `function.arguments` 是 string 类型，业务侧需要自行解析。

## 23. 推荐排查顺序

接口调用失败时，建议按以下顺序排查：

1. 检查接口地址是否正确。
2. 检查当前环境是测试环境还是现网环境。
3. 检查鉴权方式是否正确。
4. 检查 `Authorization` 或签名参数是否正确。
5. 检查 `model` 是否在当前环境支持范围内。
6. 检查 `messages` 是否符合格式要求。
7. 检查 `stream` 是否与客户端解析方式匹配。
8. 检查 `extra_body.call_from` 是否在白名单中。
9. 检查 `enable_thinking` 是否被当前模型支持。
10. 检查 function call 模型是否支持 `tools` 字段。
11. 查看 `error.message` 获取具体错误信息。
12. 使用 `session_id` 和 `interaction_id` 串联上下游日志定位问题。

## 24. 常见问题速查

### 24.1 为什么接口返回 `request_body call_from wrong`？

通常是因为 `extra_body.call_from` 不在白名单中。

处理方式：

1. 检查 `call_from` 是否填写正确。
2. 确认当前业务标识是否已配置白名单。
3. 如果只是临时联调，可先询问接口维护人应使用哪个 `call_from`。

---

### 24.2 为什么明明是思考模型，却没有传 `enable_thinking`？

`AGENT-DEEPSEEK-V3.1-THINKING` 和 `AGENT-DEEPSEEK-V3.2-THINKING` 这类模型通过模型名控制默认思考，不需要额外传 `enable_thinking`。

---

### 24.3 如何区分正文和思考内容？

流式返回中：

| 内容   | 字段                                                                  |
| ---- | ------------------------------------------------------------------- |
| 正文   | `choices[0].delta.content`                                          |
| 思考内容 | `choices[0].delta.reasoning_content` 或 `choices[0].delta.reasoning` |

如果业务只需要最终回复，只拼接 `delta.content` 即可。

---

### 24.4 流式响应什么时候结束？

通常有两个结束标志：

1. 收到：

```text
data: [DONE]
```

2. 最后一个 JSON chunk 中出现：

```json
"finish_reason": "stop"
```

---

### 24.5 非流式和流式应该选哪个？

| 场景               | 建议                      |
| ---------------- | ----------------------- |
| 前端需要边生成边展示       | 使用 `stream=true`        |
| 批处理、评测、脚本调用      | 使用 `stream=false`       |
| 业务侧暂时没有 SSE 解析能力 | 使用 `stream=false`       |
| 需要精细处理思考内容和正文    | 使用 `stream=true`，分别拼接字段 |
