# MemoryEvalUI 组内分享讲稿

## 适用场景

这份讲稿面向同组协作者，目标是让大家知道：

- 这套系统解决什么问题。
- 日常应该怎么使用。
- USER.md 和 MEMORY.md 的评测链路大概怎么跑。
- 提示词建议和闭环实验背后的核心逻辑是什么。
- 遇到失败、无候选提示词、结果不稳定时应该怎么判断。

默认分享时长：15-20 分钟讲解，5 分钟 Q&A。

## 1. 开场：为什么需要这套系统

可以这样讲：

> 我们现在要评估记忆提取结果，例如 USER.md 用户画像和 MEMORY.md 长期记忆。单靠人工逐条看成本比较高，而且 Prompt 改完之后很难稳定判断有没有变好。所以这个系统主要做三件事：把提取结果标准化成 case，用 LLM Judge 做结构化绝对评测，再基于评测证据生成候选 Prompt，形成一个可复查、可回滚的迭代闭环。

核心问题：

- 记忆提取结果不容易批量评估。
- 人工检查成本高，且难以沉淀为可复用的评测资产。
- Prompt 迭代容易凭感觉，缺少证据链。
- 同一批数据多次运行时，不只要看总分，还要看具体错误是否稳定。
- 长任务运行时间长，需要后台运行、断点续跑、限流和失败排查。

## 2. 一句话介绍系统

> MemoryEvalUI 是一个面向 USER.md 用户画像和 MEMORY.md 长期记忆的本地评测与闭环实验系统。它把记忆提取、case 生成、LLM Judge 绝对评测、结果分析、Prompt 改进建议和闭环实验串起来，用于提升记忆提取 Prompt 和评测 Prompt 的稳定迭代效率。

这里不要重点讲“页面工具”，而是讲“评测闭环系统”。

## 3. 推荐演示路线

建议按这个顺序演示：

1. **配置**
   - 填 API 地址、模型、Token、temperature、top_p、enable_thinking、并发、重试和限流等待。
   - 选择任务类型：USER.md 或 MEMORY.md。
   - 管理两类 Prompt：裁判提示词和提取提示词。

2. **记忆提取 / 数据输入**
   - 如果只有原始对话 Excel，可以在“记忆提取”里跑 USER.md 或 MEMORY.md 提取。
   - 如果已有提取结果 Excel，可以在“数据输入”里直接生成评测 case。

3. **执行评测**
   - 选择 case 文件、裁判 Prompt、提取 Prompt。
   - 启动绝对评测。
   - 评测后台运行，切换页面不会导致进度归零。

4. **结果总览 / 样本详情**
   - 总览看平均分、维度分、错误标签、失败率、稳定性对比。
   - 样本详情看对话、旧记忆、新记忆、comment、error_tags、rule_refs、evidence_refs、output_refs。

5. **提示词改进建议**
   - 基于普通绝对评测结果生成候选 Prompt 修改建议。
   - 强调只是候选版本，不自动覆盖正式 Prompt。

6. **闭环实验**
   - 自动串联：记忆提取 -> 生成 case -> 执行评测 -> 生成候选提取 Prompt -> 下一轮提取。
   - 先小样本、少轮次试跑，确认方向没跑偏再扩大。

7. **任务中心**
   - 查看后台任务状态。
   - 可以请求终止任务。
   - 运行中可调整部分安全参数，例如后续并发、请求间隔、优先级、闭环目标轮数。

## 5. USER.md 与 MEMORY.md 的边界

可以这样解释：

- **USER.md 用户画像**：偏长期稳定属性，例如身份、稳定偏好、长期交互偏好、长期关系人信息等。
- **MEMORY.md 长期记忆**：偏长期事项、计划、目标、约束、待跟踪状态和会持续更新的信息。

举例：

- “用户长期喜欢某类内容、长期偏好某种回答方式”更像 USER.md。
- “用户下周要去某地、正在规划旅行、某件事有约束和进展”更像 MEMORY.md。

讲解重点：

- 两类任务输入格式类似，但提取 Prompt 和 Judge Prompt 不一样。
- MEMORY.md 更强调事项状态、更新逻辑和长期跟踪。
- USER.md 更强调稳定画像边界，避免把临时反馈误记为长期偏好。

## 6. 普通评测链路

核心链路：

```text
原始对话 / 提取结果 Excel
    -> 标准 case
    -> LLM Judge 绝对评测
    -> EvalResult JSONL
    -> 结果总览 / 样本详情 / 提示词建议
```

评测时 Judge 会看到：

- 旧 USER.md 或 MEMORY.md。
- 对话上下文。
- 新生成的 USER.md 或 MEMORY.md。
- 裁判 Prompt。
- 可选的提取 Prompt，作为“规则来源”，不是事实来源。

Judge 输出保留字段：

- `score_total`：总分。
- `scores`：各维度分。
- `comment`：简短评语。
- `error_tags`：错误标签。
- `fatal_error`：是否严重失败。
- `diagnostics`：结构化诊断。
- `rule_refs`：违反或引用的提取规则。
- `evidence_refs`：事实证据引用。
- `output_refs`：候选输出引用。

讲解重点：

- 事实依据来自对话、旧记忆、新记忆，不来自提取 Prompt。
- 提取 Prompt 只帮助 Judge 判断“应该按什么规则评估”。
- 结果保存为 JSONL，导出的 CSV/Excel 主要用于查看，不是最完整的原始结果。

## 7. case 生成逻辑

USER.md 的核心规则：

- 先按 `轮次 == 1` 切 session。
- 每个 session 内按 `chunk_size` 分 chunk。
- 每个 chunk 调一次模型。
- 最后不足 `chunk_size` 也会调一次。
- 结果只写在当前 chunk 的最后一行。
- USER.md 会按评测人跨 session 继承。
- 如果到了应该抽取的位置但没有提取结果，会生成漏抽 case 或标记漏抽，不应该误生成完整 case。

MEMORY.md 的核心规则：

- 输入格式和 USER.md 类似。
- 结果列使用 `MEMORY.md` 或兼容列名。
- 评测人切换会重置旧 MEMORY.md。
- 同一评测人片段内部保持顺序，跨片段可以并发。

## 8. 提示词改进建议机制

这部分是同事最容易问的，建议重点讲。

### 8.1 证据从哪里来

提示词建议主要从普通绝对评测结果里收集证据。

证据字段包括：

- case_id。
- 总分和维度分。
- error_tags。
- comment。
- diagnostics。
- rule_refs。
- evidence_refs。
- output_refs。
- Judge Prompt 版本。
- 提取 Prompt 版本和 hash。

证据选择原则：

- 低分样本优先。
- 有错误标签的样本优先。
- 有 diagnostics 的样本优先。
- fatal 样本会被识别，但如果主要是调用失败或 JSON 解析失败，不能当作修改提取 Prompt 的依据。
- 如果没有明显问题样本，高分样本只作为弱上下文，不应强行修改 Prompt。

### 8.2 为什么不是让模型直接重写 Prompt

直接重写风险很高：

- Prompt 会越改越长。
- 容易针对某个 case 打补丁。
- 容易重复已有规则。
- 容易把具体人名、地点、作品名写进通用规则。
- 一次输出太长时容易超时或 JSON 解析失败。

所以系统采用“增量修改 + 安全校验”。

### 8.3 两阶段机制

提取 Prompt 优化采用两阶段：

```text
第 1 阶段：证据 -> 定位问题类型和目标章节
第 2 阶段：目标章节全文 + 同组证据 -> 生成章节级增量 patch
```

第 1 阶段：

- 优先使用评测结果里的 `rule_refs` 做本地章节定位。
- 不能本地定位的证据，再分批交给模型定位。
- 只定位，不改 Prompt。

第 2 阶段：

- 每次只给模型一个目标章节和相关证据。
- 只生成小范围 patch。
- 只允许通用规则，不允许写具体 case 补丁。
- 每条 patch 必须引用 evidence_refs。

最终还会做安全校验：

- 检查 target section 是否存在。
- 检查 evidence_refs 是否真实存在。
- 检查新增文本是否和已有规则重复。
- 检查修改比例，避免 Prompt 暴增。
- 检查冲突 patch，冲突项不自动应用。

### 8.4 候选 Prompt 怎么使用

- 候选 Prompt 会另存为新文件。
- 不覆盖原始 Prompt。
- 不自动作为正式版本上线。
- 需要人工查看 diff 和证据，再决定是否进入下一轮实验。

## 9. 闭环实验机制

闭环实验自动串联：

```text
记忆提取
  -> 生成 case
  -> 执行绝对评测
  -> 收集评测证据
  -> 生成候选提取 Prompt
  -> 用候选 Prompt 进入下一轮
```

使用建议：

- 先跑 1-2 轮。
- 每轮限制少量 case。
- 先用 mock 或小样本确认流程。
- 观察候选 Prompt 是否变长、重复、过拟合。
- 通过结果总览和样本详情做人工复核。

闭环不生成候选 Prompt 的几种情况：

| 状态 | 含义 | 建议处理 |
| --- | --- | --- |
| 评测链路失败 | 主要证据是 Judge 调用失败或 JSON 解析失败 | 先修 Judge Prompt、解析、限流或模型接口 |
| 无需修改 | 有效评测结果没有形成稳定问题证据 | 本轮可视为正常结束 |
| 无安全 patch | 有问题证据，但 patch 未通过安全校验 | 人工看证据和 diff，必要时手改 Prompt |
| 建议模型失败 | 提示词建议模型超时、断连或输出不可解析 | 减少证据条数、换模型、降低输出长度 |

## 10. 稳定性和速度策略

可以强调这些工程策略：

- temperature 默认 0。
- Judge 默认关闭 thinking，降低输出不稳定和耗时。
- Judge 输出要求结构化 JSON。
- 请求有重试和 QPS backoff。
- 同一 API/Token 使用全局请求启动间隔，避免多个功能叠加超 QPS。
- 后台任务状态落盘，切换页面不影响任务继续。
- 运行中可调整后续并发、请求间隔、优先级、闭环目标轮数。
- 结果 JSONL 边完成边保存，降低长任务失败造成的损失。
- 提示词建议使用两阶段和局部 patch，降低单次请求过长导致的失败。

## 11. 常见问题口径

### Q1：为什么总分一样，还说两次不稳定？

总分一样不代表错误分布一样。系统会看维度分、error_tags、diagnostics、comment 差异和样本级差异。两次平均分都 4.92，但具体扣分原因不同，仍然属于不稳定。

### Q2：为什么 Judge 调用失败不能拿来改 Prompt？

调用失败或 JSON 解析失败是评测链路问题，不是提取 Prompt 问题。拿它改 Prompt 会把系统故障误当成业务规则缺陷。

### Q3：为什么有时不生成候选 Prompt？

可能是两类原因：

- 有效结果显示确实没什么要改，这是正常结束。
- 有问题但证据不足、建议模型失败或 patch 不安全，此时需要人工排查。

### Q4：闭环是不是自动优化到最好？

不是。闭环只是生成候选 Prompt，并提供证据、diff 和下一轮评测结果。最终是否采用仍需要人工判断。

### Q5：为什么不让模型直接给 GSB？

当前主线已经转向绝对评测。直接让模型判胜负容易受随机性和相对偏好影响；分别绝对评分、结构化诊断和样本级对比更容易复查。

## 12. 推荐收尾

可以这样结束：

> 这套系统当前的价值不是“完全自动修好 Prompt”，而是把记忆评测和 Prompt 迭代从人工经验变成有证据、可复查、可复跑的流程。后续如果加入更强的记忆冲突检测、遗漏归因、人工反馈校准和 Agent 策略选择，它可以继续往记忆评测 Agent 或自动优化系统演进。
