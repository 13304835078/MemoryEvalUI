# MemoryEvalUI 近期改进汇报

最近主要围绕提取逻辑、评测可信度和系统稳定性做了这些改进：

- 补齐了 USER.md 和 MEMORY.md 两类任务，统一管理各自的列映射、提取 Prompt、Judge Prompt 和继承规则。提取仍按 session 和 chunk 处理，USER.md 按评测人独立继承，避免不同评测人的内容串在一起。
- 提取结果原来比较依赖固定格式，格式稍有变化就可能被当成漏抽。现在兼容 JSON、Markdown、输出标记和直接正文，并把接口、解析和 case 状态分开记录；低置信结果会保留复核，但不会继续传给下一个 chunk，避免错误扩散。
- 评测原来已经会把 `reasoning` 传给 Judge，这次主要是明确它的使用边界：reasoning 只辅助判断提取过程和更新意图，不能补充或证明用户事实，也不能替代最终输出；事实仍以旧记忆和用户对话为准，评分仍以候选结果的实际质量为主。
- Judge 还会读取实际提取 Prompt，但只把它当作规则来源。主要扣分需要给出 `rule_refs`、`evidence_refs`、`output_refs` 和结构化 diagnostics；原来可能混在 evidence_refs 中的 reasoning 引用，现在单独放到 `reasoning_refs` 做过程诊断。
- API、QPS、网络和 JSON 解析失败原来可能按 0 分进入统计。现在这些情况统一标记为未评分，与真实质量问题分开；结果页同时展示有效评分、提取覆盖率、端到端分数和运行失败数。
- 稳定性对比不再只看两轮平均分，还会比较单 case 分差、评分维度、错误标签和诊断内容。断点续跑也增加了完整指纹，样本、reasoning、Prompt、模型或解码参数变化后不会误用旧结果。
- 提示词建议从整段重写改成基于评测证据的两阶段增量修改：先定位问题和章节，再生成 replace、delete 或 append patch，并检查重复、冲突、具体 case 补丁和 Prompt 膨胀。没有生成候选时也会说明是调用失败、没有可改问题还是 patch 未通过校验。
- 闭环原来存在用同一批数据改 Prompt、再证明 Prompt 变好的问题。现在按评测人的完整历史划分 Discovery、Validation 和 Locked Test，保持 session 时序，固定 Judge 配置，并通过配对比较、退化率和 Bootstrap 置信区间决定候选是否晋升。
- 后台任务改成独立进程，并使用线程独立 HTTP Session、统一重试、跨任务限流、原子写入和增量落盘；运行中可以调整后续并发、请求间隔、优先级和闭环轮数。多人使用时按工号和姓名隔离配置、Prompt、任务和结果。
- 页面入口和结果提示重新整理，常见失败可以直接看到原因和处理建议。发布流程改成 GitHub 自动测试、打包和启动检查，目前 220 个自动化测试、11 个页面冒烟测试和开箱包启动检查均已通过。
