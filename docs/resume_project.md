## 6. 30 秒面试介绍

> 我做的是一个面向大模型长期记忆的评测和 Prompt 闭环优化系统。它支持 USER.md 用户画像和 MEMORY.md 长期记忆两类任务，可以把提取结果自动转成评测 case，用 LLM Judge 做多维绝对评分，并输出错误标签、规则引用和证据引用。基于这些评测证据，系统再生成候选 Prompt 的增量修改建议，并通过闭环实验验证下一轮提取效果。项目重点不是做页面，而是把记忆评测、结构化诊断和 Prompt 优化做成可复查、可回滚的流程。

## 7. 1 分钟面试介绍

> 这个项目背景是记忆提取 Prompt 很难稳定迭代。人工逐条看 USER.md 或 MEMORY.md 成本高，而且 Prompt 改完后很难判断到底是变好了，还是只是总分差不多。  
> 我搭建了一套评测闭环：先把原始对话或提取结果切成标准 case，再用 LLM Judge 做绝对评测。Judge 不只给总分，还要输出维度分、错误标签、comment、diagnostics，以及规则引用、事实证据引用和输出引用。这样单条扣分原因可以复核。  
> 在 Prompt 改进上，我没有让模型直接重写完整 Prompt，而是基于低分、错误标签和 diagnostics 收集证据，先定位 Prompt 章节，再生成局部增量 patch，并做重复、冲突和修改比例校验。最后闭环实验会把提取、评测、候选 Prompt 生成和下一轮提取串起来。  
> 工程上还处理了长任务后台运行、限流重试、断点续跑和稳定性对比，保证大批量评测时结果能保存、能复查、能继续迭代。

## 8. 技术追问准备

### Q1：为什么用绝对评测，而不是直接比较两个模型谁好？

直接让模型判胜负容易受相对偏好和输出顺序影响，也不利于定位错误。绝对评测会给每个 case 独立分数、错误标签和诊断，再做样本级对比，更容易复查具体问题。

### Q2：提取 Prompt 为什么也要给 Judge？

提取 Prompt 不是事实来源，而是规则来源。事实仍然来自对话、旧记忆和新记忆。Judge 参考提取 Prompt 是为了判断新记忆是否符合抽取边界，例如哪些应该记、哪些不该记。

### Q3：提示词建议的证据怎么选？

主要来自绝对评测结果：低分、错误标签、diagnostics、规则引用和证据引用。fatal error 会被识别，但如果主要是 API 失败或 JSON 解析失败，不会当成业务证据去修改 Prompt。

### Q4：为什么不让模型一次性重写 Prompt？

一次性重写容易造成 Prompt 膨胀、重复、漂移和 case 级补丁。系统把它拆成两阶段：先定位目标章节，再对单个章节生成增量 patch，并要求引用证据，最后做安全校验。

### Q5：闭环实验是不是 Agent？

当前更准确地说是 Agentic Workflow 或多阶段 LLM 编排：它串联提取、评测、诊断、候选策略生成和下一轮实验，但策略选择还比较规则化。后续可以加入更完整的 Agent 能力，例如记忆冲突检测、遗漏归因、检索增强、反思校验和人工反馈校准。

### Q6：怎么保证评测稳定？

主要从三个层面做：模型参数上 temperature=0、关闭 thinking；输出协议上要求结构化 JSON 和诊断引用；工程上做限流、重试、断点续跑、稳定性对比。稳定性对比不只看总分，还看维度分、错误标签和样本级诊断是否一致。
